# 周报生成助手

根据用户录入的周工作内容自动生成格式化工作周报。

## 功能说明

- 自动解析按日期记录的工作内容
- 智能分类整理（KWC / 低代码相关 / 项目组事务 / 技术支持 / 其他）
- 识别任务状态（重要🌟 / 完成 / 进行中）
- 简化内容描述，去除冗余信息
- 生成标准格式的周报文本

## 使用方法

1. 用户粘贴本周工作内容（按日期记录的任务清单）
2. 自动解析并生成分类整理的周报
3. 输出格式化的周报文本
4. 自动在用户当前选中的文件夹中创建周报文件

## 输入格式

用户输入通常是按日期记录的零散工作内容，格式如下：

```
2026/3/16（周一）：
1. 任务A 🌟
2. 任务B -ok
3. 任务C -?

2026/3/17：
1. 任务D
```

## 处理流程

### 第一步：内容解析

1. **日期识别**：识别 `YYYY/M/D（周X）` 格式的日期行
2. **任务提取**：提取编号列表项（1. 2. 3.）作为任务
3. **状态标记识别**：
   - 🌟 表示重要事项
   - `-ok` / `-100%` 表示已完成
   - `-?` / `-50%` 表示进行中/有阻塞
4. **子任务处理**：处理缩进的子项内容

## 第二步：任务分类

根据关键词将任务归类到以下 5 个大类，并在大类内**自动聚合生成子类别**：

**KWC**
- 关键词：KWC、组件、控件、网站、业务组件、F7、markdown、晨会、KWC开发

**低代码相关**
- 关键词：需求、验收、评审、接口对接、伙伴、客户、需求梳理、方案设计、迭代规划
- 预设子类别参考：需求协作、需求规划（可根据内容自动新增子类别）

**项目组事务**
- 关键词：补丁、hotfix、紧急修复、基线、回迁、发布说明、立项、迭代、排期、故事状态
- 预设子类别参考：补丁发布、开发计划（可根据内容自动新增子类别）

**技术支持**
- 关键词：技术支持、提单、问题处理、排查、线上问题、office插件、在线预览

**其他**
- 未匹配以上内容统一归入此项（如专利、AI体验、培训、会议等）

#### 子类别自动生成规则

1. **何时生成子类别**：当某个大类下有 **2条及以上** 任务、且任务间存在明显的主题差异时，自动按主题聚合为子类别
2. **子类别命名**：用简短的主题名称（2-6个字），如"需求协作""补丁发布""接口排查"等
3. **优先复用预设子类别**：上面列出的预设子类别优先使用，只有当任务无法归入预设子类别时才新建
4. **不强制生成**：若某大类下任务数量少（仅1条）或主题单一，直接列在大类下，不创建子类别
5. **所有大类均适用**：KWC、技术支持、其他等大类同样可以自动生成子类别

### 第三步：内容整理

1. **简化描述**：去除层级编号，精简为一句话核心描述，去掉技术细节和过程信息
2. **合并重复**：同一任务跨多天记录合并为一条，保留最新状态
3. **去除冗余**：去除【问题描述】【原因说明】【修复方案】等详细说明，仅保留结论性描述
4. **提取阻塞项**：标记为 `-?` 的任务归入"阻塞/风险"模块
5. **生成计划**：提取标记为 🌟 的重要事项作为下周计划参考

### 第四步：格式化输出

按照标准周报模板输出。大类下有子类别时使用缩进结构，无子类别时直接列任务：

```
1、本周重点进展
KWC
  网站维护              ← 自动生成的子类别
  - 任务1
  组件开发
  - 任务2

低代码相关
  需求协作
  - 任务3
  需求规划
  - 任务4

项目组事务
  补丁发布
  - 任务5
  开发计划
  - 任务6

技术支持
  文档问题              ← 自动生成的子类别
  - 任务7
  插件问题
  - 任务8

其他
- 任务9                 ← 任务少，不生成子类别

2、当前阻塞/风险/需支持事项
- 任务X（进行中）

3、下周核心计划
- 重点任务
```

## 输出格式规范

### 1、本周重点进展

#### 大类（无子类别时）
- 任务描述

#### 大类（有子类别时）
**子类别A**
- 任务描述

**子类别B**
- 任务描述

> 格式说明：
> - 大类标题独占一行，不加标点
> - 子类别用两个空格缩进，加粗显示
> - 任务列表项用 `- ` 前缀，在子类别下额外缩进两个空格
> - 无子类别时，任务直接列在大类下

### 2、当前阻塞/风险/需支持事项
- 任务描述（状态）

### 3、下周核心计划
- 计划事项

## 状态标记说明

| 标记 | 含义 | 处理方式 |
|------|------|----------|
| 🌟 | 重要事项 | 保留标记或优先展示 |
| -ok / -100% | 已完成 | 正常展示，可标注完成状态 |
| -? / -50% | 进行中 | 归入"阻塞/风险"事项 |
| （人名） | 协作人 | 保留信息 |

## 第五步：自动保存周报文件

生成周报内容后，**必须自动创建周报文件**，无需用户额外确认：

### 文件命名规则
- 格式：`YYYYMMDD-周报.md`
- 日期取**当天日期**（即生成周报时的日期）
- 示例：`20260413-周报.md`

### 保存位置
- **优先**：用户当前在 IDE 中选中的文件夹
- **回退**：如果无法确定选中文件夹，则保存到当前工作区根目录

### 文件内容
- 将生成的周报内容以 Markdown 格式写入文件
- 文件开头添加一级标题：`# 工作周报 YYYY/MM/DD`
- 其余内容与输出的周报文本一致

### 执行方式
- 使用 `create_file` 工具创建文件
- 创建完成后立即使用 `run_in_terminal` 执行 `open <文件路径>` 在 IDE 中打开该文件
- 告知用户文件已创建并打开

## 注意事项

1. 忽略具体日期，按任务类别聚合
2. 去除层级编号和多余缩进
3. 简化长描述，保留核心信息
4. 提取【问题描述】【修复方案】等关键信息
5. 同一任务跨多天记录的，合并为一条并保留最新状态
6. 生成周报后必须自动创建 `YYYYMMDD-周报.md` 文件到用户选中的文件夹，并自动打开
